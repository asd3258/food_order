## 1. 背景與目標

現況:訂餐由某人在 Teams 群組貼菜單照片,其他人用自由文字回覆想吃的餐點,無法自動統計。

目標:做一個獨立網頁(手機優先設計,之後 RWD 展開到 PC),連結分享到 Teams 聊天室即可使用(不需 Teams App 上架/Manifest,不需 SSO)。核心流程:

1. 決定今天吃哪家餐廳(單選直接開單,或多人投票決定,投票需設截止時間;可用搜尋/餐廳類型篩選快速找到餐廳)
2. 投票截止或發起者手動開票後,依票數最高的餐廳自動建立訂單
3. 開單後,每人自己選菜單品項 + 客製選項(口味/數量,類似 Uber Eats / foodpanda)
4. 系統自動統計每家餐廳訂單的品項數量與總金額(含人員),供收單/結帳用;發起者可結單或刪除訂單
5. 結單後轉入歷史訂單,發起者可勾選標記每個人的收款狀態
6. 歷史訂單留存,方便之後查詢與重複訂購

**本階段範圍**:規格 + 線框(wireframe)prototype,尚不含真正前後端串接。

**技術棄取(已確認)**:
- 前端:React 或 Vue(Mobile-first,之後用 CSS breakpoint 展開 RWD)
- 後端:FastAPI(Python)
- 部署:獨立網頁,連結貼到 Teams 聊天室使用(不做 Teams Tab/Manifest);正式環境用 Docker Compose(FastAPI + Postgres + 前端靜態網站 + MinIO 物件儲存),架在自架的 Ubuntu VM 上,見下方「開發與部署流程」
- 使用者身分:先以「可自由修改的文字欄位」模擬 Teams 使用者名稱,之後有需要再接 Teams SSO
- 「建立新餐廳」AI 解析菜單/串接 Google Maps/Uber Eats、foodpanda:**Phase 2/3,本階段只做手動輸入**
  - Phase 2:上傳菜單照片 → AI OCR 解析生成品項
  - Phase 3:貼上 Google Maps 網址 → AI 讀取店家資訊自動生成菜單草稿
  - 評估中(高風險,不承諾):串接 Uber Eats / foodpanda API 生成菜單 — 兩平台目前沒有對外開放給第三方開發者讀取任意餐廳菜單的公開 API,需先確認是否有官方合作管道,否則此項目可能無法落地。

**開發與部署流程(實際採用的做法)**:

開發在 Windows 電腦上進行,VM(Ubuntu)只負責跑正式環境的 Docker 容器,兩邊透過 GitHub 同步,不用 scp 手動搬檔案:

```bash
# 1. 在 Windows 電腦上開發、改完程式碼後推上 GitHub
git checkout main
git pull origin main          # 先同步最新版本,避免衝突
git add -A
git commit -m "說明這次改了什麼"
git push origin main

# 2. 在 Ubuntu VM 上拉下最新程式碼
cd ~/food_order
git pull origin main

# 3. 用 docker compose 重新 build + 部署(只會重 build 有變動的服務)
docker compose up -d --build
```

- 全程都直接在 `main` 分支上做,不另外切 feature 分支,避免搞混。
- 資料庫結構的異動一律走 `backend/app/migrations.py` 裡的安全 `ALTER TABLE`(見該檔案開頭說明),不用 Alembic,`docker compose up -d --build` 時會在 API 容器啟動時自動套用,不需要額外手動跑 migration 指令,也不會清空現有資料。
- 需要修改 `.env`(例如新增/更換 API 金鑰)時,只在 Ubuntu VM 上編輯,`.env` 已加入 `.gitignore`,不會被 commit、不會出現在 GitHub 上。
- 詳細的環境設定(Gemini/OpenAI/Google Places 金鑰、MinIO 密碼、防火牆開埠等)見 `README.md`。

## 2. 資訊架構(Sitemap)

底部導覽固定 4 個入口:**首頁 / 餐廳清單 / 開單與投票 / 其他功能**。

```
首頁 (Home)
├─ 使用者名稱欄位(可編輯 + Save,置於頂部,非底部導覽的一部分)
├─ 當前已發起訂單(卡片列表,點擊進入訂單明細/點餐頁)
└─ 目前投票中(卡片列表,格式同上,點擊進入投票詳情頁)
     卡片內容範例:
       投票1 餐廳名稱1/餐廳名稱2/餐廳名稱3
       發起者:Mike Chen  截止時間:2026/07/02 11:00   ← v0.5 起含年月日

餐廳清單頁 (Restaurant List) — 唯讀瀏覽
├─ 搜尋框(依餐廳名稱或品項名稱篩選,v0.5 新增)
├─ 餐廳類型篩選按鈕(便當/飲料/牛排/義大利麵,可切換,v0.5 新增)
├─ 餐廳名稱卡片列表(依搜尋/類型篩選結果顯示)
└─ 點擊 → 進入該餐廳「唯讀菜單」頁:
     ├─ 品項名稱/口味/加購/金額表格(v0.5 起為平面清單,不再依「分類」分組)
     ├─ 「✏️ 編輯餐廳資料」按鈕(v0.5 新增)→ 進入編輯餐廳頁
     ├─ 餐廳照片輪播(可左右滑動或點箭頭切換)
     ├─ 店家資訊(地址、電話、餐廳類型)
     └─ Google Maps 連結

編輯餐廳頁 (Edit Restaurant) — v0.5 新增
├─ 餐廳資料表單:名稱、電話、地址、餐廳類型(下拉選單)
├─ 餐廳照片管理:顯示現有照片縮圖,可「上傳圖片」(選取本機圖片,即時預覽)或逐張「刪除」(刪除前跳確認)
├─ 品項清單:可編輯既有品項名稱/價格,或新增/刪除品項列
└─ 「儲存變更」按鈕 → 寫回餐廳資料,回到該餐廳的唯讀菜單頁;「←」返回則捨棄變更

開單與投票頁 (Order & Vote)
├─ 搜尋框(依餐廳名稱或品項名稱篩選,v0.5 新增)
├─ 餐廳類型篩選按鈕(便當/飲料/牛排/義大利麵,可切換,v0.5 新增)
├─ 選擇餐廳(checkbox,依搜尋/類型篩選結果顯示)
└─「截止時間」選單(v0.5 起恢復年月日:日期選單/月曆 + 時 24 小時制 + 分 5 分鐘一格,預設「目前時間 +10 分」)
     ├─ 勾 1 家 → 「立即開出訂單」→ 直接建立 Order(含完整截止時間)並進入點餐頁
     └─ 勾多家 → 「進行餐廳投票」→ 建立新一批投票(含完整截止時間),導回首頁,於「目前投票中」顯示卡片

投票詳情頁 (Vote Detail) ← 從首頁「目前投票中」卡片點入
├─ 顯示截止時間(v0.5 起含年月日);若目前使用者 = 發起者,截止時間可直接用「日期選單 + 時:分」修改
├─ 候選餐廳單選 + Save/Edit(任何參與者皆可操作,行為與原規格相同)
└─ 若目前使用者 = 該投票發起者,額外顯示:
     ├─ 「開票」→ 彈出 Yes/No 確認,Yes 才依目前票數最高的候選餐廳自動建立訂單(沿用「立即開出訂單」同一套邏輯,新訂單沿用此投票的截止時間)並移除此投票,導入該訂單的點餐頁
     └─ 「刪除」→ 彈出 Yes/No 確認,Yes 才刪除此投票批次(不建立訂單),回首頁;No 則取消

訂單明細 / 點餐頁 (Order Detail)  ← 從「當前已發起訂單」點入,或由「立即開出訂單」/ 投票「開票」建立後導入
├─ 顯示截止時間(v0.5 起含年月日);若目前使用者 = 發起者,截止時間可直接用「日期選單 + 時:分」修改
├─ 若目前使用者 = 該訂單發起者,額外顯示:
│    ├─ 「結單」→ 彈出 Yes/No 確認,Yes 才將統計彙總結果 + 依人員計算的收款清單寫入歷史訂單並移除此訂單;No 則取消
│    └─ 「刪除」→ 彈出 Yes/No 確認,Yes 才刪除此訂單(不寫入歷史訂單);No 則取消
├─ 餐廳菜單品項列表(v0.5 起為平面清單,不再依分類分組)
├─ 品項 → 客製選項彈窗(口味 + 數量,類似 Uber Eats)→ 加入我的訂單(標題與卡片使用醒目色,方便使用者快速定位自己的訂單)
├─ 目前所有人統計(彙總):品項/選項、人員、數量、金額;發起者可對「其他使用者」的品項按刪除(軟刪除,畫面以刪除線標示、內容保留不移除,結單/歷史統計會排除)
└─ 底部 Google Maps 連結

其他功能 (More) — 導覽入口
├─ 歷史訂單 (Order History)
│   └─ 過去訂單列表(唯讀,品項/人員/數量/金額明細 + v0.5 新增:每人收款狀態勾選)
└─ 建立新餐廳 (Create Restaurant)
    ├─ 手動輸入:餐廳資料(含「餐廳類型」欄位,v0.5 新增)+ 品項清單(v0.5 起品項不再有「分類」欄位)
    ├─ [Phase 2] 上傳菜單照片 → AI 解析
    ├─ [Phase 3] 貼上 Google Maps 網址 → AI 生成菜單
    └─ [評估中] 串接 Uber Eats / foodpanda API 生成菜單
```

## 3. 資料模型(Data Model 草案)

| Entity | 欄位 | 說明 |
|---|---|---|
| **User** | `id`, `display_name`, `teams_name_raw`(帶入值), `updated_at` | 目前無登入機制,以 localStorage 存 `user_id`,搭配可編輯的名稱欄位 |
| **Restaurant** | `id`, `name`, `phone`, `address`, `map_url`, `type`(v0.5 新增:便當/飲料/牛排/義大利麵,單選), `created_by`, `created_at`, `updated_at`(v0.5 新增) | 餐廳清單頁顯示的基礎資料;`type` 用於搜尋頁與開單頁的類型篩選按鈕 |
| **RestaurantPhoto** | `id`, `restaurant_id`, `image_url`(v0.5:編輯頁上傳的圖片以 data URL / 檔案儲存服務網址存放), `caption`, `sort_order` | 唯讀菜單頁照片輪播用,v0.3 新增;v0.5 起可在編輯餐廳頁新增/刪除 |
| **MenuItem** | `id`, `restaurant_id`, `name`, `price`, `is_active` | 對應圖片中的品項(如「烤牛肉飯」);**v0.5:移除 `category` 欄位**,菜單改為平面清單顯示,不再依系列分組 |
| **MenuItemOption** | `id`, `menu_item_id`, `option_group`(如「口味」), `option_type`(`radio`/`checkbox`), `option_name`(如「原味/泰式」), `extra_price` | 對應圖片中「原味/泰式/五辣」等客製選項。餐廳唯讀菜單頁顯示時,依 `option_type` 收斂成兩欄:`radio` 類→口味欄,`checkbox` 類→加購欄 |
| **VoteBatch** | `id`, `initiator_user_id`, `deadline_at`(v0.5:改為完整日期時間,見下方備註), `status`(`open`/`decided`/`deleted`), `created_at` | 一次投票活動,v0.3 新增獨立實體(取代原先單一 Vote 物件),對應首頁「目前投票中」卡片 |
| **VoteBatchCandidate** | `vote_batch_id`, `restaurant_id` | 該投票批次的候選餐廳(即多選時勾選的餐廳) |
| **Vote** | `id`, `vote_batch_id`, `user_id`, `restaurant_id`(選擇的候選餐廳), `status`(`draft`/`locked`), `created_at`, `locked_at` | 單個使用者在某投票批次中的選擇;`status=locked` 對應按下 Save |
| **Order** | `id`, `restaurant_id`, `initiator_user_id`, `source_vote_batch_id`(可為 null), `deadline_at`(v0.5:改為完整日期時間), `status`(`open`/`closed`/`deleted`), `created_at`, `closed_at` | 「當前已發起訂單」的實體;若由投票開票產生則記錄來源投票批次,並沿用該投票的 `deadline_at` 作為初始值 |
| **OrderItem** | `id`, `order_id`, `user_id`, `menu_item_id`, `selected_options`(JSON), `quantity`, `note`, `is_deleted`(v0.4 新增,預設 false), `deleted_by_user_id`(v0.4 新增,可為 null), `created_at` | 每人在某訂單中選的品項(可多筆);`is_deleted=true` 代表被發起者軟刪除,前端仍顯示原內容(以刪除線標示),只是不計入統計/結單 |
| **OrderHistory** | `id`, `order_id`, `restaurant_name`, `initiator`, `closed_date`, `people_count`, `total_amount` | 結單時產生的唯讀彙總紀錄(對應歷史訂單卡片) |
| **OrderHistoryLine** | `id`, `order_history_id`, `item_label`, `user`, `quantity`, `amount` | 歷史訂單展開後的明細列(品項/人員/數量/金額) |
| **OrderHistoryPayment**(v0.5 新增) | `id`, `order_history_id`, `user`, `total_amount`, `is_paid`(預設 false), `paid_at`(可為 null) | 結單時依人員彙總金額自動產生一筆;僅該筆歷史訂單的原發起者可勾選/取消 `is_paid` |

**截止時間(deadline)規則(v0.5 恢復年月日)**:
- 記錄完整日期時間:`deadline_at`(年-月-日 時:分)
- UI 拆成三個輸入元件:原生日期選單(`<input type="date">`,即月曆選擇器)+ 時(0–23,24 小時制下拉)+ 分(僅 00/05/10/…/55,5 分鐘一格下拉)
- 建立訂單或投票時,日期預設帶入「今天」,時/分預設帶入「目前時間 +10 分」(分鐘四捨五入到最近的 5 分格)
- 目前僅作顯示與提醒用途,沒有「時間到自動觸發開票/結單」的排程機制(見第 8 節)
- v0.4 曾將此簡化為「只記當日時分、不含日期」,v0.5 依需求恢復完整日期,原因與取捨見第 8 節第 5 項(已標記為已解決)

**餐廳類型(Restaurant Type)規則(v0.5 新增)**:
- 固定四個選項:便當 / 飲料 / 牛排 / 義大利麵,單選,建立/編輯餐廳時必填
- 用於「餐廳清單頁」與「開單與投票頁」的類型篩選按鈕(可切換開關,同一時間只能選中一種類型或不選,與搜尋框篩選為 AND 關係)
- 本階段為固定清單(寫死在前端常數),尚未做成後台可管理的分類表,見第 8 節風險備註

**搜尋 / 篩選規則(v0.5 新增)**:
- 搜尋框比對「餐廳名稱」或「該餐廳任一品項名稱」,忽略大小寫,子字串比對即可(不需分詞或模糊搜尋)
- 搜尋框與類型篩選按鈕可同時套用(AND 條件),兩頁(餐廳清單頁、開單與投票頁)各自獨立記錄自己的搜尋字串與篩選狀態,互不影響

**統計邏輯**:
- 訂單統計 = `SUM(OrderItem.quantity) WHERE is_deleted=false` 依 `menu_item_id` + `selected_options` + `user_id` group by,金額 = `price + extra_price` 累加,並顯示對應人員;`is_deleted=true` 的品項在「目前所有人統計」畫面仍列出(以刪除線呈現),但不計入結單彙總與歷史訂單
- 投票統計 = `COUNT(Vote WHERE vote_batch_id=? AND status='locked')` 依 `restaurant_id` group by,`draft` 狀態不計入統計(避免還在猶豫的人被算進去)
- 開票規則(v0.3):取目前 `locked` 票數最高的 `restaurant_id`;若平票,暫定取候選清單中順序在前者(此為簡化規則,待第 8 節確認是否需要更明確的平票處理方式)
- 收款統計(v0.5 新增):結單時,依 `OrderHistoryLine` 的 `user` group by 加總 `amount`,產生對應的 `OrderHistoryPayment` 列(初始 `is_paid=false`);之後由發起者手動勾選更新,系統不做自動對帳/金流串接

## 4. 頁面規格

### 4.1 首頁(Home)

| 區塊 | 行為 |
|---|---|
| 使用者名稱欄位 | 頁面載入時嘗試帶入 Teams 顯示名稱(標準做法:分享連結時帶 `?name=` query string,由發連結的人或使用者自行輸入一次);欄位可自由修改,右側「Save」按鈕存入 localStorage + 後端 `User` 記錄 |
| 當前已發起訂單 | 列出 `status='open'` 的訂單,顯示「訂單N,發起者:XXX」+ 截止時間(v0.5 起含年月日,如 `2026/07/02 12:30`);無資料時顯示「目前沒有訂單」;點擊項目 → 導向該訂單的點餐頁(4.4) |
| 目前投票中 | 列出 `status='open'` 的 VoteBatch,卡片格式:「投票N 餐廳A/餐廳B/餐廳C」+「發起者:XXX 截止時間:YYYY/MM/DD HH:mm」;無資料時顯示「目前沒有投票」;點擊項目 → 導向投票詳情頁(4.3) |

### 4.2 開單與投票頁(Order & Vote)

此頁是唯一能建立新訂單或新投票的入口(原「餐廳清單」的選擇邏輯已整併於此;原「餐廳清單」現改為純瀏覽,見 4.5)。

**搜尋與類型篩選(v0.5 新增)**
- 頁面上方提供搜尋框,輸入餐廳名稱或品項名稱即可即時篩選下方勾選清單
- 搜尋框下方提供 4 個類型篩選按鈕(便當/飲料/牛排/義大利麵),點擊切換啟用/取消,與搜尋框篩選同時套用(AND)
- 已勾選的餐廳即使被篩選掉也**保留勾選狀態**(篩選只影響顯示,不影響已選集合),避免使用者因為改變關鍵字而不小心弄丟已勾選的餐廳

**選擇餐廳(checkbox)+ 截止時間(v0.5 恢復年月日)**
- 頁面固定顯示一組「截止時間」選單:日期(原生 `<input type="date">` 月曆選單)+ 時(0–23,24 小時制下拉)+ 分(00/05/10/…/55,5 分鐘一格下拉),進入頁面時日期預設「今天」,時/分預設帶入「目前時間 +10 分」
- 勾選 **1 家** → 按鈕文字「立即開出訂單」,點擊直接建立 `Order(status='open', deadline_at=目前選單值)`,導向該訂單的點餐頁(4.4)
- 勾選 **多家** → 按鈕文字「進行餐廳投票」,點擊建立 `VoteBatch(status='open', deadline_at=目前選單值)` + 對應 `VoteBatchCandidate`,清空選擇,導回首頁 — 該投票會出現在首頁「目前投票中」卡片列表
- 兩種情況都必須有截止時間,但因選單永遠有預設值,不需要額外的「未填」檢查

### 4.3 投票詳情頁(Vote Detail)

- 標題顯示「投票N」
- **截止時間**(v0.5 起含年月日):
  - 一般參與者:純文字顯示「截止時間 YYYY/MM/DD HH:mm」
  - 若目前使用者 = 發起者:改為顯示「日期選單 + 時/分下拉選單」(值為目前 `deadline_at`)+「更新」按鈕,點擊後直接覆寫 `VoteBatch.deadline_at`
- 候選餐廳單選 list + 票數顯示,**Save/Edit** 行為與 v0.2 相同:
  - **Save**:鎖定當前選擇(選項反灰、不可再點選其他選項),寫入 `Vote(status='locked')`
  - **Edit**:解除反灰狀態,允許重選,但**不會**立即改變後端資料,要等使用者再按一次 Save 才會覆寫記錄
  - 驗收要點:Edit 之後如果使用者中途放棄離開頁面(沒再按 Save),系統應保留「上一次 Save 的結果」
- **若目前使用者 = 該投票發起者**,額外顯示兩個按鈕,**兩者皆為點擊後先彈出 Yes/No 確認視窗,選 Yes 才真正執行、選 No 則直接關閉視窗、不做任何事**:
  - **開票**(Yes 後執行):依第 3 節的開票規則找出目前票數最高的候選餐廳,直接建立 `Order`(等同「立即開出訂單」的建單邏輯,`deadline_at` 沿用此投票原本的值),並將此 `VoteBatch.status` 設為 `decided`(從首頁列表移除),導入新訂單的點餐頁
  - **刪除**(Yes 後執行):將此 `VoteBatch.status` 設為 `deleted`(從首頁列表移除),不建立任何訂單,回首頁
- 目前版本沒有「非發起者可否刪除/開票」的限制討論,預設僅發起者可操作,詳見第 8 節風險備註

### 4.4 訂單明細 / 點餐頁(Order Detail)

- **截止時間**(v0.5 起含年月日):與投票詳情頁邏輯相同——一般使用者看純文字「截止時間 YYYY/MM/DD HH:mm」;發起者看到「日期選單 + 時/分下拉選單」+「更新」按鈕,可直接修改 `Order.deadline_at`
- **若目前使用者 = 該訂單發起者**,頁面上方額外顯示兩個按鈕,**皆為點擊後先彈出 Yes/No 確認視窗,選 Yes 才執行、選 No 直接關閉**:
  - **結單**(Yes 後執行):將「目前所有人統計(彙總)」的結果(排除已被軟刪除的品項)整理成歷史訂單明細(品項/人員/數量/金額),並依人員加總金額產生對應的收款清單(`OrderHistoryPayment`,初始未收款),一併寫入歷史訂單,並將此訂單從「當前已發起訂單」移除
  - **刪除**(Yes 後執行):直接將此訂單從「當前已發起訂單」移除,**不**寫入歷史訂單
- 進入後顯示該訂單對應餐廳的菜單品項列表(**v0.5 起為平面清單**,不再依分類分組——因 MenuItem 已移除 `category` 欄位)
- 點擊品項 → 彈出客製選項彈窗(bottom sheet,仿 Uber Eats / foodpanda):
  - 口味/加購選項(單選或多選,依 `option_group` 決定 UI 是 radio 還是 checkbox)
  - 數量(+/- stepper)
  - 「加入我的訂單」按鈕 → 寫入一筆 `OrderItem`
- 「我的訂單」標題與卡片使用醒目色(暖色系,區別於品牌主色),方便使用者在長清單中快速找到自己已加入的品項與小計,可移除/修改數量
- 訂單發起者可看到「目前所有人統計(彙總)」表格,欄位為:**品項 / 選項、人員、數量、金額**(每一列對應一筆 OrderItem,不再合併同品項多筆記錄,方便逐筆刪除)
  - **軟刪除其他使用者的品項**:發起者在「人員」不是自己的資料列上會看到「刪除」連結;點擊後彈出 Yes/No 確認,Yes 才將該筆 `OrderItem.is_deleted` 設為 true——畫面上該列會保留原內容,但整列加上刪除線樣式,且不計入結單彙總/歷史訂單;自己加入的品項則透過上方「我的訂單」卡片的「移除」功能處理(直接從清單移除,非軟刪除)
- 頁面最下方提供 **Google Maps** 連結,點擊開啟該餐廳地圖(以店名+地址組成搜尋網址)

### 4.5 餐廳清單頁(Restaurant List)— 唯讀瀏覽

- **搜尋與類型篩選(v0.5 新增)**:頁面上方提供搜尋框(依餐廳名稱或品項名稱篩選)+ 4 個餐廳類型篩選按鈕(便當/飲料/牛排/義大利麵,可切換),兩者為 AND 關係,即時套用到下方名稱卡片列表
- 顯示已建立餐廳的**名稱卡片**(無 checkbox,純瀏覽用途,與 4.2 的選餐廳邏輯分開),卡片副標顯示餐廳類型
- 點擊卡片 → 進入該餐廳的唯讀菜單頁,由上到下依序為:
  1. **菜單表格(v0.5 起為平面清單,不再依「系列」分類分組)**,欄位為:**品項名稱、口味、加購、金額**
     - 口味欄:彙整該品項所有單選型(radio)選項群組的選項名稱,如「原味/泰式/五辣」;無則顯示「無」
     - 加購欄:彙整該品項所有多選型(checkbox)選項群組,格式「選項名稱+加價金額」,如「白飯加量20/半熟蛋15/泡菜20」;無則顯示「無」
  2. **「✏️ 編輯餐廳資料」按鈕(v0.5 新增)**:點擊進入 4.6 編輯餐廳頁
  3. **餐廳照片輪播**:多張照片以橫向卡片呈現,可用手指滑動切換,或點擊左右箭頭按鈕切換(對應 `RestaurantPhoto`,依 `sort_order` 排序)
  4. **店家資訊**:地址、電話、餐廳類型
  5. **Google Maps 連結**:點擊開啟該餐廳地圖
- 此頁**不提供點餐功能**,純供查詢/確認菜單內容用,要開單或投票請到「開單與投票」頁(4.2)

### 4.6 編輯餐廳頁(Edit Restaurant)— v0.5 新增

- 由 4.5 餐廳唯讀菜單頁的「✏️ 編輯餐廳資料」按鈕進入,進入時將該餐廳資料深拷貝一份草稿(`editingRestaurant`),避免使用者按「←」放棄編輯時污染原始資料
- **餐廳資料表單**:名稱、電話、地址、餐廳類型(下拉選單,固定 4 個選項)
- **餐廳照片管理**:
  - 以縮圖 grid 呈現目前草稿中的所有照片(含原有的示意圖與已上傳的真實照片)
  - 「+ 上傳圖片」開啟本機檔案選擇器,選取圖片後透過 `FileReader.readAsDataURL()` 轉為 data URL 即時預覽並加入草稿(本階段 wireframe 用瀏覽器記憶體暫存,正式版需改為上傳到檔案儲存服務並存網址)
  - 每張縮圖右上角有「×」刪除鈕,點擊先跳 Yes/No 確認,Yes 才從草稿中移除該張照片
- **品項清單**:列出草稿中的品項(名稱 + 價格可直接編輯),可「+ 新增品項」新增一列空白品項,或於每列點「刪除」移除該品項(v0.5 簡化:編輯頁的品項僅管理名稱與價格,客製選項群組的編輯留待後續版本)
- **儲存變更**:將草稿整份寫回 `restaurants` 陣列(取代原資料),提示成功後導回該餐廳的唯讀菜單頁
- **放棄變更**:點擊左上「←」直接捨棄草稿,回到原餐廳唯讀菜單頁,原始資料不受影響

### 4.7 其他功能(More)— 導覽入口頁

集中放置非日常高頻操作的功能入口,底部導覽第 4 個分頁:
- **歷史訂單**(4.8)
- **建立新餐廳**(4.9)

### 4.8 歷史訂單頁(Order History)

- 列出已結單的訂單,唯讀
- 每筆卡片顯示:餐廳、發起者、日期、參與人數、總金額、**收款進度(v0.5 新增,如「已收款 2/4」)**;展開後列出:
  1. 明細表格,欄位為:**品項、人員、數量、金額**
  2. **收款狀態區塊(v0.5 新增)**:依人員列出「姓名 + 應付金額 + 已收款 checkbox」;**僅該筆歷史訂單的原發起者**可勾選/取消勾選(勾選即時切換 `is_paid`,無需額外確認,方便隨手記帳);非發起者檢視時 checkbox 顯示為停用(disabled)、僅供查看
- 明細資料來源:訂單發起者按下「結單」時,依 4.4 節統計彙總邏輯產生(品項+選項+人員 group by);收款清單則依人員加總金額自動產生

### 4.9 建立新餐廳頁(Create Restaurant)

**本階段(Phase 1)**:純手動輸入表單
- 餐廳基本資料:名稱、電話、地址、Google Map 連結、**餐廳類型(v0.5 新增,下拉選單,必填)**
- 品項清單(可動態新增列):名稱、價格(**v0.5:移除「分類」欄位**)
- 每個品項的客製選項(可動態新增群組,如「口味」→ 原味/泰式/五辣)

**Phase 2(規劃保留 UI 入口,本階段不實作)**:
- 上傳菜單照片 → 呼叫 AI(如 Claude API)做 OCR + 結構化解析,自動產生上述品項與選項草稿,使用者再手動校正

**Phase 3(規劃保留 UI 入口,本階段不實作)**:
- 貼上 Google Maps 網址 → AI 讀取該店家的公開資訊(店名、菜單照片、評論等)自動生成菜單草稿,使用者再手動校正
  - 風險備註:Google Maps 對第三方的資料讀取有使用政策限制,實作前需確認透過 Google Places API 等官方管道取得資料的合法性與涵蓋範圍(店家是否真的把完整菜單放在 Maps 上,不一定,常只有照片)

**評估中(不承諾)**:
- 串接 Uber Eats / foodpanda API 自動生成菜單 → **高風險項目**,兩平台皆無公開的第三方讀取 API,建議列為「待評估」而非承諾功能

## 5. 訂單/投票狀態機

```
投票批次流程 (VoteBatch):
  open (投票進行中,任何參與者可 Save/Edit 自己的選擇;發起者可隨時修改 deadline_at)
    → 發起者按「開票」→ [Yes/No 確認] → Yes → decided (依票數最高的候選餐廳建立 Order,deadline_at 沿用,從首頁移除)
    → 發起者按「刪除」→ [Yes/No 確認] → Yes → deleted (從首頁移除,不建立訂單)
    (No → 停留在 open,不改變任何狀態)

單個投票紀錄流程 (Vote,屬於某個 VoteBatch):
  draft (使用者選擇中,未鎖定)
    → Save → locked (計入統計,選項反灰)
    → Edit → draft (解除反灰,可重選;尚未覆寫後端資料)
    → 再次 Save → locked (以新選擇覆寫)

訂單流程 (Order):
  open (可繼續點餐 / 修改;可能由「立即開出訂單」或投票「開票」建立;發起者可隨時修改 deadline_at,可對其他人的品項按刪除)
    → 發起者按「結單」→ [Yes/No 確認] → Yes → closed (統計彙總 + 依人員產生收款清單,寫入歷史訂單,排除軟刪除品項,從首頁移除)
    → 發起者按「刪除」→ [Yes/No 確認] → Yes → deleted (直接從首頁移除,不寫入歷史訂單)
    (No → 停留在 open,不改變任何狀態)

OrderItem 軟刪除流程:
  is_deleted=false (正常計入統計)
    → 發起者對非本人品項按「刪除」→ [Yes/No 確認] → Yes → is_deleted=true (畫面保留、以刪除線標示,不計入統計/結單)
    (No → 保持 is_deleted=false)

OrderHistoryPayment 收款狀態流程(v0.5 新增):
  is_paid=false (結單時自動產生,預設未收款)
    → 原發起者勾選 checkbox → is_paid=true (paid_at 記錄時間)
    → 再次取消勾選 → is_paid=false (paid_at 清空)
    (非發起者無法操作,checkbox 顯示為 disabled)
```

## 6. API 草案(FastAPI)

| Method | Path | 說明 |
|---|---|---|
| GET | `/api/users/me` | 依 localStorage 帶的 user_id 取回顯示名稱 |
| PUT | `/api/users/me` | 更新顯示名稱 |
| GET | `/api/restaurants?q=&type=` | 餐廳清單(唯讀瀏覽用);v0.5 新增 `q`(名稱/品項關鍵字)與 `type`(餐廳類型)查詢參數 |
| POST | `/api/restaurants` | 手動建立餐廳(含餐廳類型、品項與選項) |
| GET | `/api/restaurants/{id}/menu` | 取得該餐廳菜單品項 + 客製選項 + 照片 + 地址電話 + 類型 |
| PUT | `/api/restaurants/{id}` | v0.5 新增:編輯餐廳資料(名稱/電話/地址/類型/品項) |
| POST | `/api/restaurants/{id}/photos` | v0.5 新增:上傳一張餐廳照片 |

## 7. RWD 策略

- 開發順序:先做手機尺寸(以 390px 寬為基準),用 flex/grid + rem 單位,不寫死 px
- Breakpoint 建議:`≤600px` 手機(單欄)、`601–1024px` 平板(雙欄,清單+明細並排)、`>1024px` 桌機(三欄:導覽 + 清單 + 明細/統計側欄)
- 手機版點餐用「品項 → bottom sheet 彈窗」;桌機版可考慮改成右側 side panel,而不強制彈窗,但本階段先以彈窗元件為主,PC 版重用同一元件即可,先不特化
- 照片輪播(4.5):手機版為橫向滑動單張全寬顯示;桌機版可考慮改為多張並排的 grid,不強制輪播,待實作時依畫面寬度決定
- 搜尋框 + 類型篩選按鈕(v0.5):手機版垂直堆疊(搜尋框在上、篩選按鈕列在下,可換行);桌機版可考慮並排在同一列,待實作時依畫面寬度決定

## 8. 待確認 / 風險

1. 使用者身分:目前無 Teams SSO,同一人若換裝置/清 localStorage 會被視為新使用者 — 是否可接受先用「名字純文字比對」當作暫時身分?
2. **開票平票規則**(v0.3 新增):目前 wireframe 簡化為「取候選清單中順序在前者」,正式版是否需要更明確的規則(如重新投票、發起者手動決定)?
3. **權限範圍**(v0.3 新增,v0.4 部分解決):「開票/刪除投票」「結單/刪除訂單」「軟刪除他人品項」現在點擊後都會先跳 Yes/No 確認視窗,降低誤觸風險;但目前仍僅限發起者操作,是否需要允許管理員或其他角色代為操作,尚待確認。**v0.5 新增同類問題**:「編輯餐廳資料」「切換收款狀態」目前也沒有角色限制之外的權限模型(編輯餐廳任何人都可操作,收款狀態僅限該筆訂單發起者),是否需要更細緻的權限(例如僅限建立餐廳的人可編輯)待確認
4. **投票截止時間的自動化**(v0.3 新增):目前只能由發起者手動按「開票」,截止時間目前僅作為顯示/提醒用途(發起者可隨時修改),尚未實作「時間到自動開票/自動結單」的排程機制,是否需要?
5. ~~**截止時間不含日期**~~ **(v0.4 提出,v0.5 已解決)**:v0.4 曾簡化為「當日 HH:mm」,v0.5 已恢復完整年月日,支援跨日情境(例如晚上先開明天中午的投票)
6. **軟刪除的通知機制**(v0.4 新增):發起者刪除其他人的品項目前沒有另外通知被刪除的人(只在畫面上看到自己的品項被劃掉),是否需要跳出提示或送通知給對方,待確認
7. Uber Eats / foodpanda API 串接:高風險,建議正式排入路線圖前先做技術可行性確認
8. Google Maps 資料串接(Phase 3 建立餐廳、4.4/4.5 的地圖連結):目前地圖連結僅用「店名+地址」組成搜尋網址,建議正式版改用 Google Place ID 以提高準確度;Phase 3 的 AI 自動生成菜單需先確認 Google Places API 的資料涵蓋範圍與使用政策
9. 多人同時編輯同一張投票/訂單的資料衝突處理(本階段用「最後寫入為準」,無樂觀鎖),量大後可能需要加版本號機制
10. **收款狀態的正確性與提醒**(v0.5 新增):收款狀態純靠發起者手動勾選,系統不驗證金流,也沒有提醒未收款人員的機制;是否需要加入到期提醒(如訂單結單 N 天後仍有未收款,提示發起者催款)待確認
11. **餐廳類型為固定清單**(v0.5 新增):目前「便當/飲料/牛排/義大利麵」四種類型寫死在前端常數,尚無後台可管理的分類表;若餐廳種類擴增(如飲料店同時賣輕食),單選是否足夠、是否需要改為多標籤(tags),待確認
12. **餐廳照片上傳的儲存與大小限制**(v0.5 新增):目前 wireframe 用 `FileReader.readAsDataURL()` 在瀏覽器記憶體暫存圖片,正式版需要串接實際的檔案儲存服務(如物件儲存)並限制檔案大小/格式,以及決定圖片是否需要伺服器端壓縮
13. **移除品項「分類」後的資訊密度**(v0.5 新增):部分餐廳品項較多時,平面清單(不分類)可能造成單頁項目過長,是否需要在資料模型不變的前提下,以其他 UI 方式(如可收合區塊、依價格排序)輔助瀏覽,待確認
